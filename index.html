IoT-MOS API INTEGRATION GUIDE

IoT-MOS exposes sensor data through two complementary channels: an HTTP REST API for querying and an MQTT API for streaming. Pick whichever fits your use case, or use both together. Every reading is organized under the same registered structure, whichever channel you read it from.
Integration model

Pull · on demand
HTTP REST

You ask, we answer. Ideal for reports, dashboards, and historical queries.

  • ProtocolHTTPS (TLS 1.2+)
  • PatternClient pulls on demand
  • AuthJWT
  • FreshnessAs of last query
  • Best forReports, history, dashboard reloads
Push · real-time
MQTT

We push the moment something changes. Ideal for live displays and alerts.

  • ProtocolMQTT 3.1.1 / 5.0 over TLS
  • PatternBroker pushes on event
  • AuthX.509 client certificate
  • FreshnessSub-second, event-driven
  • Best forFloor maps, occupancy, alerts

HTTP rate limit — strict

The HTTP REST API is limited to one request every 15 seconds per tenant. Requests beyond this are rejected with 429 Too Many Requests. If you need a faster refresh, that’s a real-time use case — use the MQTT channel instead.

Use HTTP for query-shaped access: a daily occupancy report, a BI tool, a dashboard that refreshes on a click, or a sensor’s last-known status at app startup.

Use MQTT to react the instant something changes: a live screen of vacant booths, push alerts, digital signage. One persistent connection replaces thousands of polls.

Use both (hybrid) for most production deployments: subscribe to MQTT for live state, and call HTTP on startup (to warm caches) and on demand (for any past window).

Choosing the channel

Three questions settle most decisions:

Do you need to see changes within one second of them happening?
If yes → MQTT
Do you need data from before “now” — yesterday, last week, a custom range?
If yes → HTTP
Both of the above?
→ Hybrid — MQTT for live, HTTP (status history) for the past
