Channels

Pull · on demand

HTTP REST

You ask, we answer. Ideal for reports, dashboards, and historical queries.

  • ProtocolHTTPS (TLS 1.2+)
  • PatternClient pulls on demand
  • AuthJWT
  • FreshnessAs of last query
  • Best forReports, history, dashboard reloads

Push · real-time

MQTT

We push the moment something changes. Ideal for live displays and alerts.

  • ProtocolMQTT 3.1.1 / 5.0 over TLS
  • PatternBroker pushes on event
  • AuthX.509 client certificate
  • FreshnessSub-second, event-driven
  • Best forFloor maps, occupancy, alerts

HTTP rate limit — strict

The HTTP REST API is limited to one request every 15 seconds per tenant. Requests beyond this are rejected with 429 Too Many Requests. If you need a faster refresh, that is a real-time use case — use the MQTT channel instead.

When to use which

HTTP when you need query-shaped access: a daily occupancy report, a BI tool, a dashboard that refreshes on a user click, or a sensor's last-known status at app startup.

MQTT when you need to react the instant something changes: a live screen showing vacant lockers, push alerts, digital signage. One persistent connection replaces thousands of polls.

Both (hybrid) for most production deployments: subscribe to MQTT for live state, and call HTTP on startup (to warm caches with last-known values) and on demand (for any past time window). No polling, full depth, comfortably within the rate limit.

HTTP REST API

The REST API is fully documented as an interactive OpenAPI (Swagger) reference. Browse the endpoints, inspect request and response schemas, and try calls live from the page.

Authentication is via JWT, obtained by POST. Remember the 1-request-per-15-seconds limit when designing refresh behavior.

03

MQTT API

The MQTT channel streams readings in real time from AWS IoT Core. It is documented as an AsyncAPI specification — the messaging-world equivalent of OpenAPI — which renders as an interactive reference and defines every topic, parameter, and payload schema.

ASYNCAPI Open the MQTT API reference
Hosted alongside the REST docs — channels, topic parameters, and payload schemas ↗

Link to be finalized once the AsyncAPI page is published to GitHub Pages.

The reference covers the full topic structure, subscription wildcards, X.509 authentication and tenancy, payload schemas, and how to test a subscription. You connect to AWS IoT Core with an X.509 client certificate bound to your companyId; the broker enforces tenant isolation, so you only receive topics under your own company.

04

Choosing your channel

Three questions settle most decisions:

Do you need to see changes within one second of them happening?
If yes → MQTT
Do you need data from before "now" — yesterday, last week, a custom range?
If yes → HTTP
Both of the above?
→ Hybrid — MQTT for live, HTTP (status history) for the past
Questions?

This guide is a starting point — explore the two references and reach out to IoT-MOS API Support with any questions about onboarding, certificates, or designing your integration.
